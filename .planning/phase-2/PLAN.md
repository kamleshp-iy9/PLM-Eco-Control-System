# Phase 2 Plan — Odoo-Inspired PLM Feature Enhancement

**Phase goal:** Implement the full Odoo PLM feature set on top of the existing React + Express + Prisma foundation, delivering enterprise-grade ECO governance, BoM versioning with visual diff, effectivity date scheduling, document version management, conflict detection, work center master data, and a data-rich analytics dashboard.

**Priority order (highest to lowest business value):**
1. Per-stage approval governance with approval matrix (stage-scoped approvers, COMMENT_ONLY type, rejection-revert)
2. Effectivity date scheduling with ASAP / date modes
3. BoM version history browser with side-by-side diff and one-click restore
4. Work center master data + enriched BoM operations
5. Document versioning on Product and BoM attachments
6. ECO conflict detection and soft-lock
7. Product lifecycle status transitions (Draft / Active / End-of-Life / Obsolete)
8. Enhanced analytics dashboard (cycle time, throughput, top contributors)

#### 3b. `advanceEcoStage` rewrite (eco.service.js)

The existing logic checks global `ApprovalRule` records. Replace with stage-scoped logic:

```
When action = 'APPROVE':
  1. Load ApprovalRule records where stageId = currentStage.id AND isActive = true
  2. Load EcoApproval records for this ECO where action = 'APPROVED'
  3. For REQUIRED rules: check if req.user has a REQUIRED rule for this stage
     - If yes: create EcoApproval record (action=APPROVED, stageId stored in comment JSON or separate field)
     - Check if ALL REQUIRED approvers for this stage have now approved
       - If all approved: advance to nextStage
       - If not all approved: return eco at current stage with "Approval recorded. Waiting for N more approver(s)."
  4. For OPTIONAL rules: create approval record, advance immediately
  5. For COMMENT_ONLY: create approval record (action=APPROVED), do NOT advance

When action = 'VALIDATE' (no approval required):
  - Current behavior preserved: advance immediately
```

#### 3c. Rejection with revert (eco.service.js + eco.controller.js)

Current `rejectEco` sets state to `CANCELLED`. Replace with:

**New `rejectEcoStage`** function:
```
1. Create EcoApproval record (action=REJECTED, comments required)
2. Find the previous stage (sequence = currentStage.sequence - 1)
3. If previous stage exists: set eco.stageId = previousStage.id (revert, NOT cancel)
4. If no previous stage (currently at stage 1): set eco.state = CANCELLED
5. Create AuditLog: action='APPROVAL_REJECTED', oldValue={stage: currentStage.name}, newValue={stage: previousStage.name OR 'CANCELLED'}
```

Add route `POST /api/ecos/:id/revert` for manual engineer-initiated revert (no approval record, just stage regression).

**Files to modify:**

### What to build

#### 4a. ECO creation / update changes

`createEco` body now accepts:
- `effectivityType: 'ASAP' | 'SCHEDULED'` (default `ASAP`)
- `scheduledDate: ISO8601 string` (required when `effectivityType = 'SCHEDULED'`)

Validation: if `effectivityType = 'SCHEDULED'`, `scheduledDate` must be in the future.

Store both fields on the `Eco` record (schema added in Task 1).

#### 4b. `applyEco` service change

At the end of `applyEco` (after all Prisma writes):
```javascript
if (eco.effectivityType === 'ASAP') {
  // Existing behavior: apply immediately, set effectiveDate = now
  await tx.eco.update({ where: { id: ecoId }, data: { isApplied: true, state: 'APPROVED', effectiveDate: new Date() } });
} else {
  // SCHEDULED: mark as pending_activation, do NOT archive old BoM/Product yet
  await tx.eco.update({ where: { id: ecoId }, data: { state: 'APPROVED', isApplied: false, effectiveDate: eco.scheduledDate } });
  // The actual BoM/Product version swap happens in effectivity.service.js on scheduledDate
}
```

The SCHEDULED path skips the BoM/Product archive step — it writes `proposedChanges` to the DB but defers the version creation.

#### 4c. `effectivity.service.js` — scheduled activation

```javascript
const activatePendingEcos = async () => {
  const now = new Date();
  // Find approved ECOs with SCHEDULED effectivity where scheduledDate <= now and isApplied = false
  const pending = await prisma.eco.findMany({
    where: {
      effectivityType: 'SCHEDULED',
      state: 'APPROVED',
      isApplied: false,
      scheduledDate: { lte: now },
    },
    include: { bom: { include: { components: true, operations: true } }, product: true, stage: true },
  });

  for (const eco of pending) {
    await applyScheduledEco(eco);  // calls the BoM/Product version creation logic
  }
};
```

Use `node-cron` (already in many Express stacks; add to `package.json` if absent) to schedule `activatePendingEcos` every 5 minutes:
```javascript
// In server/src/app.js or a dedicated cron.js imported by app.js
const cron = require('node-cron');
cron.schedule('*/5 * * * *', activatePendingEcos);
```

Add a manual trigger endpoint for testing: `POST /api/ecos/trigger-scheduled-activation` (ADMIN only).

#### 4d. New API endpoint

**`GET /api/ecos/pending-scheduled`** — returns all ECOs with `effectivityType=SCHEDULED, isApplied=false, state=APPROVED` with their `scheduledDate`. Used by the admin dashboard to show upcoming activations.

### Acceptance criteria
- Creating an ECO with `effectivityType=ASAP` and approving it applies the BoM version immediately (existing behavior preserved)
- Creating an ECO with `effectivityType=SCHEDULED, scheduledDate=<future>` and approving it sets `state=APPROVED, isApplied=false`
- `GET /api/ecos/pending-scheduled` returns the scheduled ECO
- Calling `POST /api/ecos/trigger-scheduled-activation` at/after `scheduledDate` applies the BoM version and sets `isApplied=true`
- Passing `scheduledDate` in the past when `effectivityType=SCHEDULED` returns 400 validation error

## Task 8 — Product Lifecycle Status API

**Files to modify:**
- `server/src/controllers/product.controller.js` — add lifecycle transition handler
- `server/src/routes/product.routes.js` — add `POST /:id/lifecycle` route

### What to build

#### 8a. `POST /api/products/:id/lifecycle`

```javascript
const transitionLifecycle = async (req, res, next) => {
  const { id } = req.params;
  const { toStatus, reason } = req.body;  // toStatus: 'DRAFT'|'ACTIVE'|'END_OF_LIFE'|'OBSOLETE'

  const product = await prisma.product.findUnique({ where: { id } });
  if (!product) throw new NotFoundError('Product not found');

  // Validate allowed transitions:
  const ALLOWED = {
    DRAFT: ['ACTIVE'],
    ACTIVE: ['END_OF_LIFE'],
    END_OF_LIFE: ['OBSOLETE', 'ACTIVE'],  // can reactivate
    OBSOLETE: [],                          // terminal state
  };

  if (!ALLOWED[product.lifecycleStatus].includes(toStatus)) {
    throw new ValidationError(`Cannot transition from ${product.lifecycleStatus} to ${toStatus}`);
  }

  // ADMIN can bypass lifecycle for corrections (req.query.force='true')

  const [updatedProduct, history] = await prisma.$transaction([
    prisma.product.update({ where: { id }, data: { lifecycleStatus: toStatus } }),
    prisma.productLifecycleHistory.create({
      data: { productId: id, fromStatus: product.lifecycleStatus, toStatus, changedBy: req.user.id, reason },
    }),
  ]);

  await createAuditLog({ action: 'LIFECYCLE_TRANSITION', entityType: 'Product', entityId: id, oldValue: { lifecycleStatus: product.lifecycleStatus }, newValue: { lifecycleStatus: toStatus }, userId: req.user.id });

  res.json({ success: true, data: { product: updatedProduct, history } });
};
```

#### 8b. `GET /api/products/:id/lifecycle-history`

Return all `ProductLifecycleHistory` records for the product, ordered `createdAt DESC`, include `user: { select: { name } }`.

#### 8c. `POST /api/products/bulk-lifecycle` (ADMIN only)

Accept `{ productIds: string[], toStatus, reason }` and apply the same transition to all products in a single transaction. Return `{ updated: N, failed: [{ productId, error }] }`.

### Acceptance criteria
- New product has `lifecycleStatus = 'DRAFT'`
- `POST /api/products/:id/lifecycle` with `{ toStatus: 'ACTIVE' }` on a DRAFT product succeeds
- `POST /api/products/:id/lifecycle` with `{ toStatus: 'DRAFT' }` on an ACTIVE product returns 400
- `GET /api/products/:id/lifecycle-history` returns the transition records
- `POST /api/products/bulk-lifecycle` with 3 DRAFT products → returns `{ updated: 3, failed: [] }`

## Success Criteria for Phase 2

The phase is complete when all of the following are measurably true:

1. **Schema integrity** — `npx prisma migrate dev` completes on the existing DB with no data loss to existing ECOs, products, BoMs, or users.

2. **Work center master data** — ADMIN can create, edit, deactivate, and delete work centers at `/settings/work-centers`. Work center is selectable on BoM operations.

3. **Per-stage approval governance** — A stage with 2 REQUIRED approvers requires both approvals before the ECO advances. Rejection reverts to the prior stage. COMMENT_ONLY approvals record without advancing. The EcoDetailPage shows "N of M required approvals received."

4. **Effectivity scheduling** — ECOs with `effectivityType=SCHEDULED` do not apply their BoM changes until the cron job fires at/after `scheduledDate`. ASAP ECOs behave exactly as Phase 1.

5. **BoM version history** — `/master-data/boms/:id/versions` shows a timeline of all versions with color-coded inline diffs. Restore creates a new ECO pre-populated with the old version's components.

6. **Document versioning** — Products and BoMs support file uploads with automatic version increment when the same filename is re-uploaded. Upload history is visible per filename.

7. **Conflict detection** — Two simultaneous BOM ECOs on the same BoM are blocked at create time. An amber warning banner appears on ECO detail when a softer conflict exists (same product, different ECO types).

8. **Product lifecycle** — Products have a `lifecycleStatus` field cycling DRAFT → ACTIVE → END_OF_LIFE → OBSOLETE. Transitions are validated, logged, and visible on the product detail page.

9. **Analytics dashboard** — `/analytics` shows ECO throughput (12-month bar chart), lifecycle distribution (donut), top contributors, and approval bottleneck table, all sourced from live data.

10. **Reports CSV export** — All three existing report tabs (ECOs, Product Versions, BoM Changes) have a Download CSV button that triggers a file download.
